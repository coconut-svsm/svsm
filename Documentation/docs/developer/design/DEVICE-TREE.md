Device Tree
===========

COCONUT-SVSM uses a Flattened Device Tree (FDT) to communicate which devices
are assigned to the SVSM firmware plane. This is necessary for devices such as
virtio-MMIO whose base addresses have no standard discovery path visible to
SVSM. The device tree is generated by the hypervisor at VM creation time and
delivered to SVSM as an IGVM parameter. SVSM parses the FDT at boot to
discover virtio-MMIO transports and other platform devices without relying on
ACPI or firmware configuration interfaces.

This document specifies the device tree bindings used by SVSM. Where
applicable, bindings follow the conventions established by the [Devicetree
Specification v0.4](https://github.com/devicetree-org/devicetree-specification/releases/tag/v0.4)
and the [Linux kernel devicetree documentation](https://www.kernel.org/doc/Documentation/devicetree/bindings/).

### Delivery mechanism

The FDT blob is passed to SVSM as an IGVM parameter area, populated by the
hypervisor via the IGVM `DeviceTree` parameter directive.

### Device plane filtering

Devices may carry a `plane` property indicating which VM plane (VMPL)
owns them. SVSM includes a device if the `plane` property is absent or equals
`SVSM_VMPL` (0), and ignores it otherwise.

The `plane` property is currently optional to retain compatibility with
hypervisors that do not yet emit it.

Bindings
--------

### Root node

Follows the [Devicetree Specification v0.4 §3.2](https://github.com/devicetree-org/devicetree-specification/releases/tag/v0.4)
with the addition of the `device-plane` property.

| Property         | Type   | Required | Description                                |
|------------------|--------|----------|--------------------------------------------|
| `compatible`     | string | yes      | Must be `"coconut,svsm"`.                  |
| `#address-cells` | u32    | yes      | Number of cells to encode child addresses. |
| `#size-cells`    | u32    | yes      | Number of cells to encode child sizes.     |
| `device-plane`   | u32    | yes      | The VMPL the guest OS runs at.             |

### virtio-mmio

Virtio memory-mapped I/O transport nodes. Follows the Linux
[virtio/mmio.yaml](https://www.kernel.org/doc/Documentation/devicetree/bindings/virtio/mmio.yaml)
binding with the addition of a `plane` property.

**Node name:** `virtio_mmio@<hex-address>`

| Property     | Type     | Required | Description                                                                    |
|--------------|----------|----------|--------------------------------------------------------------------------------|
| `compatible` | string   | yes      | Must be `"virtio,mmio"`.                                                       |
| `reg`        | u64, u64 | yes      | Base address and size of the MMIO region.                                      |
| `plane`      | u32      | no       | The VMPL that owns this device. When absent it is treated as `SVSM_VMPL` (0). |
| `interrupts` | u32      | no       | Interrupt specifier.                                                           |

**Differences from the Linux binding:** The Linux binding requires
`dma-coherent`. SVSM does not manage DMA coherence at this layer. The `plane` property
is an SVSM-specific extension (see [Device plane filtering](#device-plane-filtering)).

Example
-------

```dts
/dts-v1/;

/ {
    compatible = "coconut,svsm";
    #address-cells = <0x02>;
    #size-cells = <0x02>;
    device-plane = <0x02>;

    virtio_mmio@fef00000 {
        compatible = "virtio,mmio";
        reg = <0x00 0xfef00000 0x00 0x200>;
        plane = <0x00>;
    };

    virtio_mmio@fef01000 {
        compatible = "virtio,mmio";
        reg = <0x00 0xfef01000 0x00 0x200>;
        plane = <0x00>;
    };
};
```
